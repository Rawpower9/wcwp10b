**From Directory to Dialogue**  
As a current AI major and aspiring software engineer/researcher, I want to further understand how software development and AI research currently operates. But the way these jobs operate are constantly changing, even more so today. One of the largest reasons for the change today is the use of generative AI for writing code. Since generative AI can generate vast amounts of code very quickly, the way engineers and researchers interact with code has changed drastically, which can be seen by the difference in the tools used today compared to even just 5 years ago.   
What prompted me to explore this space was just seeing the headlines every month. Sometimes I would see “Software development is dead”, while other times I would see “The end of coding”. There’s no denying that students with Computer Science degrees fresh out of college are struggling to find jobs. In addition, many large companies are cutting a large number of software engineering jobs. The reason why, as I’m sure you’ve guessed at this point, is cited as AI integration. So, just by looking at all of this, I thought that by the time I actually get my degree, the profession that I hope to occupy one day would be taken over by AI. So I wanted to find more information about what was actually happening.   
Before generative AI developer tools were created, some of the most popular tools for development, called Integrated Development Environments or IDEs for short, were VS Code, Jet Brains and Atom (Singh). But all three of these IDEs have one thing in common: their layout. These tools always had a file directory to the left, the main coding window in the center, and sometimes a terminal at the bottom. Each tool came packaged with its own host of features, but these tools all had the same purpose: to help a developer write code.  
Those tools now seem outdated compared to the tools used today. Tools that are more popular today include Cursor, Codex and Claude Code. When I first opened Cursor, I was pretty confused. While it did look very similar to VS Code, there was one big thing missing: the file directory. The entire panel was replaced with a chat window to interact with a coding agent and an agents list. The first thing I thought of when I was Cursor was “how can I program if I can’t see any of my files?” Claude Code, one of the most popular software development tools today, doesn’t even have a window to see code\! Claude Code is just a terminal window with a way to interact with coding agents. That’s it.

	The first question I had when I saw this was why are the programming environments so vastly different, but that question was quickly answered. Since AI is faster, and sometimes better, at programming, a software developer no longer needs the ability to write code in these new environments. But that led me to a second, and arguably more important question, which became my essential question: **If software engineers aren’t writing code, then what are they currently doing?** This question is also at the crux of the tension that inspired me to explore this space.  
	At first it seems obvious: They just prompt the AI to write a piece of code or complete a feature for them. But that means being a developer is very simple. If software engineers are prompting a large language model to write code for them, which is something anyone can do, then are they really needed anymore? Does the transition from an IDE centered around writing code to using tools centered around writing prompts mark the end of software engineering?

	As I dove deeper into the issue, I found more and more evidence marking the end of software engineering. For instance, people without any prior programming experience wrote a few prompts, and got a fully fledged application in return. Currently, there are even whole apps solely built around the idea that anyone can create an app. But many people that use those apps say that after a certain point, it takes way more prompts to get the AI to implement a new feature. That’s because “\[building a\] personal laundry app is very different from the enormously complex software that drives businesses and governments. … Building these applications requires the planning, guidance and experience of \[experienced\] coders” (Metz). After apps get sophisticated enough, AI agents require more technical expertise so that they don’t make mistakes in the process of creating a new feature.

	Does that mean that the job of a software engineer is to guide AI agents in building sophisticated tools? Well, yes and many of the sources that I read only reinforce this idea. In a New York Times article about the current state of Artificial Intelligence, two seasoned software engineers went through the process of creating a word editor like google docs, completely from scratch. While the engineers estimated that they “would have needed at least two months to build this complex piece of software \[on their own\]. With Codex, they finished in two days” (Metz). But they also reported something else. Because of the complexity of the project, they needed to read over the code that it was generating in order to ensure that no mistakes were being made.

	The same thing is also true for engineers without as much experience. A student here at UCSD, Vikram, augments his work with the use of AI agents because it’s not only faster, but it also means he doesn’t have to do the tedious part of software development. Vikram works on developing software for autonomy in Triton Robotics, a robotics club on campus, localization for underwater robots and many hackathon projects. While in the past, he could have done all by painstakingly sifting through youtube videos for tutorials, blogs to fix errors and manually writing every line. However now, he offloads a large part of the code that he has to write to Claude Code. But he’s still responsible for guiding his AI agents through the software that he needs.

	So what does this mean? Currently, software engineers across the board use tools incorporating AI agents to write code, but the engineers are still responsible for guiding the AI agents through the process, fixing mistakes when they happen and ensuring that the code created is actually the kind of software that they need. According to a study on how AI is used in Software Engineering done in 2024, 68% of the engineers in the study used AI in at least one part of their workflow. They also found that engineers that used Github Copilot, an AI coding assistant that autocompletes and generates code, completed tasks 55% faster than engineers that didn’t. But this study was done in 2024, and as AI tools get better it will boost productivity further leading to more software engineers adopting AI development practices.  
	As I conducted my research, I noticed something startling: AI generated code is not always better. For instance, a 2024 report from the Center for Security and Emerging Technologies stated that while AI tools are adept at developing software that seems to work, it also can incorporate unintended cybersecurity risks. Your personal data isn’t all stored on your computer, a lot of it is stored in servers somewhere in the cloud. Applications that have your personal data with cybersecurity vulnerabilities risk having hackers gain access to the server that it is hosted in, meaning that they could access your personal information.  
	Not only is this bad for a user, this is also bad for companies, so why risk it? And this isn’t the only issue. As I continued my research, I found studies that indicated that AI written code is easy to break. Yet with all of its issues, these tools are being widely used by companies. I wanted to understand why, and understand the underlying systems. After doing some research, I found that it’s because they have to. According to a survey by Zapier, an automation platform, a majority of companies feel pressured to use AI tools in order to stay competitive with their competitors. Since using AI tools makes building software faster, companies that don’t use it risk falling behind. So software engineers that don’t use AI tools could actually pose a bigger risk to a company than AI generated code\!  
	To summarize my findings, trying to figure out why there was a change in programming environments, from the IDEs of the past to tools integrating AI agents of today, uncovered a shift in the way software is developed as a whole. It’s not an understatement to say that this is a paradigm shift away from manually writing code to just writing prompts. Yet this paradigm shift also uncovers a change in what it means to be a software engineer.

	Instead of trying to write code to fix problems or create features, software engineers are now just having an AI agent write code. There are many reasons why, but the main reasons are that it’s faster and easier to do. But that means their role has shifted from being a programmer to being a guide. But that begs the question, if the constant use of AI allows software engineers to become effective guides. In order to be a good guide, they have to go through the process, understand the pitfalls and be ready to catch others if they go down the wrong path. But will software engineers without a lot of experience be able to effectively guide AI agents? If not, then how will aspiring software engineers like myself be able to hold the current software infrastructure up after the current experienced ones retire? I don’t have the answer to those questions, but the current development landscape of quickly generating software speeds up productivity today, but may, in the future, come back to bite us.
